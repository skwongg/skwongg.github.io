Object Oriented Programming

February 19, 2015

In the following, I hope to discuss two different approaches to programming which are Object Oriented Programming and Functional Programming. Both languages are very capable of producing programs that are functional and efficient. The argument of speed, maintainability and so forth in regards to either being superior are typically in extreme use cases or not universally true. Instead of thinking in terms of black or white, I'd like to take the discussion to be within the continuum in between, I'd like to steer the discussion towards... the 50 or so shades of grey.

From the research in which I conducted, neither Object Oriented Programming nor Functional Programming are required to be completely mutually exclusive of each other. Thus far I have had the most experience with ruby and despite it being an object-oriented language, I can see functional components and methodologies that could work well in ruby. I think that recognizing the differences within the context of the programmer's abilities is more important than pointing out contextual differences. There are many online posts that suggest one programming style to be superior to the other for specific use cases, but never a poster who claims that the other style can't accomplish the same thing.

Object oriented programs are typically built from the top down, starting with the class, variables, methods and so forth to build the details necessary. Object oriented programming is well suited for applications that can capitalize on its prebuilt object/class structures. If the purpose of the application is to organize, repackage and repurposify data (as with many websites we use daily!), object oriented programming may be convenient because the framework is already built-in.

Functional programming on the other hand builds the program up from the bottom up, building small functions that can be combined into other functions to solve larger problems. Additionally, since functional programming generally emphasizes NOT modifying things, you can typically implement earlier pieces of code seamlessly since their functionality is more or less static (unchanged/unmodified).

Next we have style, the paradigm in which your code follow determine what principle nature your code follows. In Functional Programming, it is taboo to rename or redefine functions, instead you create new functions and store them. And instead of appending value indexes etc., there is an inclination towards creating a new value to reflect the changes.



Example1

Object Oriented Programming
dog = ["lassy", "spot", "blue"]
dog << ["dawg"]

Functional Programming
dog = ["lassy", "spot", "blue"]
dogs = dog + ["dawg"]



Example2

Object Oriented Programming:

hash = {:potatoes => "browns", :yolo => "sweggerella"}
hash[:bro => "swol"]
hash




Functional Programming:

hash = {:potatoes => "browns", :yolo => "sweggerella"}
new_hash = hash.merge(:bro => "swol")

Though the differing programming types result in the same output, they represent different perspectives on code. Object oriented programming is very convenient for its predefined classes, methods, instance variables and capturing the interaction between these things by calling methods on them. Since nearly everything in ruby is an object, programming in ruby simply becomes creating methods to appropriately connecting and referencing the objects.

Methods in ruby in which we have become familiar such as map, select, inject, standard if/elsif/else statements and a handful of other ruby techniques are accepted as the functional parts of ruby. The conflict between object oriented programming and functional programming paradigms lies in changing states of objects, the changing (appended/modified) variables, and complexity of syntax. As of currently, I am not well-versed in either to pick a side, so with that I bid you adieu!
